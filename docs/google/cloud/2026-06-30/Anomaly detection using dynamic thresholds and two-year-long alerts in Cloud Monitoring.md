# Anomaly detection using dynamic thresholds and two-year-long alerts in Cloud Monitoring

- **來源**: Google Cloud Blog
- **發布日期**: 2026-06-30
- **原文連結**: https://cloud.google.com/blog/products/management-tools/cloud-monitoring-adds-long-lookback-alert-policies-for-promql/

## 核心主題
Google Cloud Monitoring 推出支援 **兩年回顧**（two-year lookback）的 PromQL 警報策略，讓開發者能以動態閾值取代固定閾值，根據指標的歷史資料自動調整警報基準，解決傳統靜態閾值無法應對時變負載與成長型工作負載的痛點。

## 關鍵重點
- **動態閾值取代固定閾值**：傳統靜態閾值面臨三大問題——工作成長後閾值過低導致誤報頻繁、時變指標（如電商交易量依一天中時段變化）無法用單一閾值捕捉、每種負載需建立獨立策略。新功能允許 PromQL 警報策略查詢最多 **兩年** 的歷史資料，實現「如果過去五分鐘的平均值是過去一周平均值的兩倍，就觸發警報」這類相對基準的動態閾值。
- **三種動態閾值演算法**：①**移動平均**：適合相對穩定的資料，當近期趨勢偏離長期移動平均 30% 時觸發。②**Z-score（標準分數）**：基於平均值與標準差衡量統計距離，超過 3 或低於 -3 即視為異常，適合平均穩定且具足夠波動性的資料。③**季節性分解（時間偏移比較）**：將資料與前一天或前一周的同一時段比較，適合有時間模式變化的指標（如網站流量依時段/日變化）；提醒使用時間偏移時應只追蹤下降或上升其中之一，避免 24 小時後出現「假性異常」的雙重警報。
- **防止雲服務費用失控的實用範例**：以 AI token 成本為例，可設定動態閾值警報——當最近 10 分鐘的 token 使用量超過一周歷史平均的 **25 倍**時，觸發 Pub/Sub 通知至 Cloud Run function，再透過 Cloud Quotas API 將 Token Usage 配額降至 0，立即止血。Google 同時宣布正在產品化動態閾值的異常偵測功能，並開發專為時間序列資料訓練的 AI 模型。目前開放申請設計夥伴（preview partner）。

## 結論
Cloud Monitoring 的兩年回顧動態閾值功能，將警報從「我設定一個數字，超過就報」升級為「根據你自己的歷史行為，什麼算異常」。這不僅解決了成長型與時變工作負載的監控痛點，更打開了自動化成本控制的實用場景——當 AI token 用量異常飆升時，系統能自動偵測並強制停損，讓 SRE 團隊從被動救火轉為主動防禦。
