# Is it agentic enough? Benchmarking open models on your own tooling

- **來源**: HuggingFace Blog
- **發布日期**: 2026-06-18
- **原文連結**: https://huggingface.co/blog/is-it-agentic-enough

## 核心主題
Hugging Face 團隊提出「代理使用測試」概念——程式庫不應只是正確且快速，還必須設計成讓代理能有效驅動。透過 transformers 案例研究，團隊建立了 `agent-eval` 基準測試工具，測量代理完成任務所需的「工作成本」，而非僅僅檢視最終答案是否正確。

## 關鍵重點
- **所有成功不等同，需測量整個過程**：兩個代理都能給出情感分類的正確答案，但一個需寫 40 行 Python 除錯、另一個只需一行 CLI 命令。傳統基準只看最終字串，卻對這些差異與程式庫改進（如 CLI、Skill）是否真正幫助代理毫無知覺。團隊建立包含匹配率、中位時間/ Token、錯誤率與「標記」（markers，如是否調用 CLI、是否使用 pipeline() API）的多維度分數系統。
- **CLI 對大型模型有益、對小型模型有害的反直覺發現**：Kimi-K2.6、GLM-5.1 等強模型在 Skill 層級利用 CLI 以更少回合數完成任務；但小型模型傾向依賴訓練數據中的 memorized API 模式，新概念反而增加犯錯面積——Qwen3-4B 在 Clone 層級新增 Token 從 ~2.4k 飆升至 ~23k（10 倍）且準確率無提升；Qwen3-14B 在 Skill 層級情感分類從 100% 暴跌至 0%（代理誤將 CLI 視為可調用工具）。
- **工具設計與實用指引**：評估分為三種層級（bare：僅 pip install；clone：克隆原始碼；skill：載入打包文件與範例）。大型模型應「固定模型、變更修訂」測量工作量；小型模型應「固定修訂、遍歷模型」。團隊提供 `agent-eval` CLI 工具，可在 HF Jobs 上並行輸出實驗報告，並建議「強模型生成 Skill、小模型預先驗證」的 Upskill 流程作為代理友好工具的標準優化方法。

## 結論
若沒有這套基準測試工具，Hugging Face 維護者可能會憑直覺合併一個「加速大型模型卻破壞小型模型」的提交——同一個改動對不同大小的代理產生截然不同的效果。核心教訓是：面向代理的 API 必須跨模型大小評估，代理時代的軟體開發需要「如果沒測試，它就不存在；如果沒文件，它就不存在」的新思維。
